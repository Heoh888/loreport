# Доменные понятия

## Lore

Lore — это накопленный контекст проекта:

- почему что-то существует;
- почему было принято конкретное решение;
- какие зависимости или клиенты завязаны на текущую реализацию;
- что нельзя удалять без последствий.

Это не просто markdown-документ и не статическая wiki-страница. В терминологии проекта lore — это живая инженерная память.

## Sidecar

Loreport задуман как sidecar-сервис рядом с основным приложением:

- монтирует тот же репозиторий;
- пишет результаты обратно в `loreport/`;
- живёт в одном stack'е с app, DB и queue;
- не заменяет исходный сервис, а наблюдает за ним.

## Snapshot

Snapshot — это проверка, изменилось ли содержимое каталога `loreport/` после agent-run.

Назначение:

- определить, был ли реальный результат;
- не обновлять metadata при no-op;
- избежать бесконечных циклов sync.

## Update metadata

`REPO_PATH/loreport/.last-update.json` хранит информацию о последнем успешном обновлении.

По коду и документам в metadata ожидаются:

- `updatedAt`
- `command`
- `model`
- `gitHead` или близкое поле

## Git evidence

Agent-контекст собирается из git:

- `git status --short`
- `git rev-parse HEAD`
- `git log` с последнего sync или последние 20 коммитов
- `git diff --name-status HEAD`

Это помогает агенту видеть текущее состояние репозитория, а не работать вслепую.

## Providers и модели

Backend поддерживает несколько провайдеров через конфиг:

- OpenRouter
- OpenAI
- Anthropic
- другие, предусмотренные в плане и constants

Секреты лежат только в env vars. Настройки провайдера и model id — не секретные и могут храниться отдельно.

## Sync job

Sync job — единица работы, которая проходит через API, очередь, worker и PostgreSQL.

Состояния из типов:

- `pending`
- `running`
- `done`
- `failed`

Для UI они сворачиваются в более простую модель отображения статуса.

## Virtual filesystem paths

Agent использует виртуальные пути вроде:

- `/README.md`
- `/loreport/quickstart.md`

Это часть портирования из OpenWiki и важный контракт для prompt logic.
